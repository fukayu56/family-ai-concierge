# AI Development Playbook

全AIアプリ共通の開発ルール集。

Family AI Concierge に限らず、今後開発するすべてのAIアプリで共通利用する。
単なるコーディング規約ではなく、設計思想・AIへの依頼方法・品質基準まで含むプレイブックである。

---

## 1. Purpose

このドキュメントの目的は次のとおり。

- AIアプリ開発で品質を維持する
- AI（Cursor / Claude Code / Codex）へ一貫した指示を出す
- 設計を優先した開発を行う

個別プロダクトの方針（例：開発憲章）がある場合は、**プロダクト憲章を優先**し、本プレイブックは共通の開発作法として併用する。

---

## 2. Development Philosophy


| 方針           | 意味                          |
| ------------ | --------------------------- |
| まず設計         | 実装の前に、責務・入出力・画面フロー・成功条件を決める |
| 小さく作る        | 1回の変更はレビューできる単位に制限する        |
| 動くものを早く確認する  | 仮説検証のために、早く動く状態をつくる         |
| MVPを最優先する    | 完成条件に含まれないものは後回しにする         |
| 将来拡張できる設計にする | 過剰実装はしないが、拡張を塞がない形にする       |
| コードより設計を重視する | 「動くコード」より「正しい境界と契約」を優先する    |


迷ったときは「この変更は設計を明確にするか？ MVPを前進させるか？」で判断する。

---



## 3. Design Principles

このPlaybook全体で共通となる設計思想。

### 基本原則

- シンプルさを優先する（Simple is Better）
- ユーザー価値を最優先する
- 将来変更しやすい設計にする
- 1つのコンポーネント・Context・関数は1つの責務だけ持つ
- コードは「書く時間」より「読まれる時間」の方が長いことを意識する
- 共通化は「3回以上同じコードが現れたら」検討する
- MVPでは過度な最適化をしない
- まず正しく動く最小構成を作り、その後に保守しやすい形へ整える
- AIが実装しやすい構造を意識する
- 人が読んで理解しやすいコードを書く

迷った場合は、よりシンプルで責務が明確な設計を採用する。

---



## 4. AI Collaboration Rules

Cursor / Claude Code / Codex などへ依頼するときのルール。

### 依頼する側

- 実装前に設計（目的・変更範囲・非目標・DoD）を確認する
- 一度に大きく変更しない（画面・API・スキーマを無関係にまとめて変えない）
- プロダクト憲章・本プレイブックに反する要望を出さない



### 実装する側（AI）

- 既存機能を壊さない
- 変更理由を書く
- 差分（何を・なぜ・どこを）説明する
- TypeScriptエラー0を維持する
- 必要以上にリファクタリングしない
- MVP外の機能は実装せず、提案のみにとどめる
- ルールに反する場合は実装せず、理由を説明する
- 提案は理由付きで行う

### 実装後の報告（必須）

実装後は必ず以下を報告する。

- Charterとの整合性
- Playbookとの整合性
- Guardrails違反の有無
- 変更ファイル一覧
- 変更理由
- 影響範囲
- TypeScriptエラー有無
- 今回のDoD達成状況

### 依頼テンプレート（推奨）

1. 目的
2. 変更対象ファイル / 非対象
3. 受け入れ条件（DoD）
4. 触ってはいけない既存挙動
5. 今回はやらないこと（Non Goals）

---



## 5. AI Guardrails

AI（Cursor / Claude Code / Codex）が勝手に設計を変更しないためのガードレール。

AIは以下を勝手に実施しない。

- `package.json` を変更しない
- 新しいライブラリを追加しない
- ディレクトリ構成を変更しない
- Contextの責務を変更しない
- Router構成を変更しない
- API仕様を変更しない
- JSON Schemaを変更しない
- Validationルールを変更しない
- TypeScriptの型を削除しない
- UIデザインを大きく変更しない
- 命名規則を変更しない
- 開発憲章（Project Charter）を書き換えない
- Playbookを書き換えない
- MVP外の機能を勝手に追加しない

変更が必要だと判断した場合は、次を説明したうえで**提案**として提示し、ユーザーの承認を得てから実装する。

1. なぜ変更が必要なのか
2. メリット
3. デメリット
4. 影響範囲
5. 代替案

## 6. Architecture Rules

責務を混ぜない。どこに何を書くかを固定する。


| 層                       | 責務             | 書いてよいもの                   | 書いてはいけないもの                        |
| ----------------------- | -------------- | ------------------------- | --------------------------------- |
| **Screen**              | 画面表示とユーザー操作の入口 | UI状態、画面内ハンドラ、遷移           | APIキー、ビジネスルールの本体                  |
| **Component**           | 再利用可能なUI部品     | propsに基づく表示・操作            | 画面固有フロー、グローバル状態の直接操作（必要な場合は明確に制限） |
| **Context**             | 画面間で共有する状態     | 状態・永続化・更新関数               | API呼び出しの詳細実装、UIレイアウト              |
| **Router**              | 画面遷移の経路        | ルート定義、レイアウト               | ドメインロジック                          |
| **API（クライアント）**         | サーバーへの通信       | fetch、URL、リクエスト送信         | 機密キーの保持、レスポンスの穴埋め正規化で不正データを通すこと   |
| **API（サーバー）**           | 外部AI/外部サービスの窓口 | バリデーション、プロンプト構築、外部呼び出し、ログ | クライアント専用UI前提のコード                  |
| **Validation / Schema** | 契約の定義と検証       | 型、JSON Schema、validate関数  | 画面文言、ビジネス判断の全て                    |




### 原則

- **単一の真実の源泉**: 同じ型・同じスキーマを二重定義しない（必要なら共有する）
- **外部境界で検証する**: AIレスポンス・外部APIレスポンスは必ず検証してから内部へ渡す
- **依存の向き**: UI → Context / API Client → Server → AI/外部API

---



## 7. Data Source Policy

外部データは必ず SpotCandidate に正規化してから利用する。

AIは外部データの形式（CSV、JSON、GeoJSONなど）を直接扱わない。

新しいデータソースを追加するときは、
SpotService または専用Mapperのみを変更する。

Prompt、Schema、Validationは変更しない。

---



## 8. Prompt Design Rules

AIへ送るプロンプトは、感覚ではなく契約として設計する。

必ず含める要素:

1. **役割** — モデルが何者として振る舞うか
2. **目的** — 何を達成するか
3. **出力形式** — JSONなど、機械可読な形式
4. **考慮事項** — 制約・優先順位・リスク
5. **回答ルール** — 件数、言語、禁止事項
6. **JSON Schema** — 返すべき構造の正
7. **Validation** — Schemaと照合する実行時チェック
8. **Structured Outputs** — 可能な場合はモデル側でも Schema に縛る



### 運用ルール

- JSON以外の文章を返させない（説明は構造内のフィールドに書く）
- Markdownコードブロックを付けさせない
- 項目名の変更・省略を禁止する
- 値が未取得の外部情報は断定させない
- プロンプトと Schema と Validation の3点を揃える

---



## 9. AI Model Policy

| フェーズ | 方針 |
|---|---|
| 開発時 | 高速・低コストモデルを利用する |
| 品質検証 | 本番候補モデルで比較する |
| 本番 | コストと品質を考慮して決定する |

- モデル名をコードにベタ書きしない
- 環境変数または設定ファイルから変更できるようにする

---



## 10. UI / UX Rules

- 入力は最小限にする（初期値・再利用で作業を減らす）
- 状態が分かりやすい（読込中・成功・失敗を明示する）
- 編集画面を分離する（入力と閲覧の役割を混ぜない）
- 一覧は読むだけにし、編集は編集画面へ誘導する
- 長い画面はスクロール対応する
- レスポンシブ対応する（少なくともスマホ操作を想定する）
- ボタン配置・サイズは押しやすさを優先する
- ユーザーの操作回数を減らす
- エラー時は「次に何をすればよいか」を示す
- AI処理中はローディング状態を表示する

---



## 11. API Rules

- Request と Response を型定義する
- Validationを必ず行う（特にAIレスポンス）
- JSON Schemaを維持し、実装と一致させる
- エラー時は理由を返す（例: `{ "error": "..." }`）
- ログを残す（開始・成功・失敗。秘密情報は出さない）
- APIキーはクライアントに置かない
- 外部AI呼び出しはバックエンド経由に限定する
- 検証に失敗したデータは画面へ渡さない

---



## 12. TypeScript Rules

- 型安全を最優先する
- `any` は禁止（やむを得ない場合は理由と範囲を明示する）
- Context には型を書く
- 型を使い回す（同じ形状を複数箇所で別定義しない）
- Optional を適切に使う（本当に欠ける値だけ `?`）
- ランタイム検証と型定義を可能な限り揃える
- TypeScriptエラーを残したままマージしない

---



## 13. Error Handling

エラーは種類ごとに扱いを分ける。


| 種別               | 開発者向け                 | ユーザー向け                  |
| ---------------- | --------------------- | ----------------------- |
| 通信エラー            | ネットワーク失敗・タイムアウトをログ    | 通信失敗と再試行手順を示す           |
| Validationエラー    | どの契約違反かをログ            | 形式不正であることと再試行/戻る操作を示す   |
| APIエラー           | status / サーバーエラー内容をログ | 分かりやすい失敗理由を示す           |
| OpenAI / 外部AIエラー | プロバイダエラーをサーバーでログ      | 一般化した失敗メッセージ（内部詳細は出さない） |


原則:

- ユーザー向け表示と開発者向けログを分ける
- 秘密情報（APIキー、個人情報の過剰出力）をログに出さない
- 失敗時は不正データをUIへ伝播させない

## 14. Development Workflow

推奨フロー:

```text
設計
  ↓
画面
  ↓
Context
  ↓
API
  ↓
Validation
  ↓
AI
  ↓
改善
```

各段階の最低条件:

1. **設計** — 目的、入出力、Non Goals、DoDを決める
2. **画面** — 操作フローと状態表示をつくる
3. **Context** — 共有状態と保存方針を決める
4. **API** — Request/Response型とエラー契約を決める
5. **Validation** — Schemaと検証関数を揃える
6. **AI** — プロンプト + Structured Outputs + 検証
7. **改善** — 実機確認結果から小さく直す

後ろの工程から始めない。AI接続は、契約（型・Schema・Validation）が固まってから行う。

---



## 15. Git Rules

- コミットは小さくする
- 意味が分かる単位で分ける（機能 / 修正 / ドキュメント）
- レビューしやすい差分にする（無関係な整形を混ぜない）
- 秘密情報をコミットしない
- メッセージは「なぜ」が分かるように書く

---



## 17. Non Goals

今やらないこと / MVPでやらないことを、毎回明確にする。

共通の Non Goals 例:

- 早すぎる最適化
- 認証・決済・高度な履歴管理（プロダクト憲章で後回しとされたもの）
- 完璧なデザインやストア公開対応の先取り
- 根拠のない大規模リファクタリング
- Schema未定義のまま進めるAI連携
- 「将来使うかも」だけでは足りない抽象化

プロダクト固有の Non Goals がある場合は、その開発憲章を正とする。

---



## 18. Definition of Done

完成とみなす条件:

- [ ] TypeScriptエラー0
- [ ] 画面確認済み
- [ ] Context動作確認済み
- [ ] API確認済み
- [ ] Validation確認済み
- [ ] JSON Schema確認済み
- [ ] 不要な `console.log` 削除
- [ ] 不要コメント削除
- [ ] 開発憲章違反なし
- [ ] 既存機能の退行がない

いずれかが欠ける場合は未完成として扱う。

---



## 19. AI Checklist



### 依頼前

- [ ] 目的と受け入れ条件が書けている
- [ ] 変更対象 / 非対象が分かれている
- [ ] Non Goals が明示されている
- [ ] 開発憲章・本プレイブックに矛盾していない
- [ ] 一度に変更する範囲が大きすぎない



### 実装後（AIの返答レビュー）

- [ ] 変更理由と差分説明がある
- [ ] TypeScriptエラー0
- [ ] 既存機能を壊していない
- [ ] MVP外を勝手に実装していない
- [ ] Validation / Schema / 型が揃っている
- [ ] エラー時のユーザー導線がある
- [ ] 不要ログ・不要コメントがない
- [ ] DoDを満たしている
- [ ] 実装後の必須報告がある（Charter整合性 / Playbook整合性 / Guardrails違反の有無 / 変更ファイル一覧 / 変更理由 / 影響範囲 / TypeScriptエラー有無 / 今回のDoD達成状況）

## 共通利用の宣言

**この Playbook は、今後開発するすべてのAIアプリで共通利用する。**

個別プロダクトの Mission / Vision / 憲章はプロダクトごとに持つ。
一方で、設計の進め方・AI協業・品質基準は本ドキュメントを共通の基準とする。
